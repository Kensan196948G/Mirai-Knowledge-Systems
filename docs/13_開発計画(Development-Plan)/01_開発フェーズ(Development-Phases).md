# 開発フェーズ策定

## 前提
- プロトタイプ開発環境 → 本番環境の順で策定する
- Docker等の仮想環境は使用しない
- 各フェーズの期間は未規定 (決定後に追記)
- 承認は依頼者が実施、レビュー評価(数値)は本ドキュメントに基づき実施

## レビュー評価(数値化)の方針
- 各フェーズ完了時に以下の観点を1〜5点で評価し、合計25点で記録する
- 観点: 完成度 / 一貫性 / 実現性 / リスク / 受入準備
- 目安: 20点以上で次フェーズに進行、20点未満は是正対応

## 1. プロトタイプ開発環境の詳細開発フェーズ (全8フェーズ)

### フェーズ1: スコープ・課題定義
- 目的: プロトタイプで検証すべき課題と利用シナリオを確定
- 実施内容:
  - 主要ユーザー(施工管理/品質/安全)の課題整理
  - 検証対象ユースケースの選定
  - 画面範囲の確定
- 成果物:
  - スコープ定義メモ
  - 対象ユースケース一覧
- レビュー評価の観点:
  - 課題の網羅性、ユースケースの妥当性

### フェーズ2: 情報設計(IA)・ナビゲーション設計
- 目的: 画面構成と情報優先度を確定
- 実施内容:
  - 画面構成図とナビゲーション設計
  - 主要カード/モジュールの配置方針
- 成果物:
  - 画面構成図
  - ナビゲーション設計メモ
- レビュー評価の観点:
  - 情報の優先順位、迷いなく操作できる構造

### フェーズ3: ワイヤーフレーム作成
- 目的: 主要画面のレイアウトと情報量の妥当性を検証
- 実施内容:
  - ダッシュボード/詳細画面のワイヤー作成
  - 表示密度と視線導線の確認
- 成果物:
  - 主要画面ワイヤー
- レビュー評価の観点:
  - 情報量の適正、視認性

### フェーズ4: ビジュアル設計・コンポーネント定義
- 目的: UIトーンとコンポーネントの共通化
- 実施内容:
  - 色/タイポ/余白ルール定義
  - バッジ、カード、リストの設計
- 成果物:
  - UI方針/コンポーネント一覧
- レビュー評価の観点:
  - 統一感、視認性、現場用途への適合

### フェーズ5: 静的WebUI実装(HTML/CSS)
- 目的: プロトタイプとして操作感のベースを構築
- 実施内容:
  - `webui` へのHTML/CSS実装
  - レスポンシブ対応
- 成果物:
  - 画面HTML/CSS一式
- レビュー評価の観点:
  - デザイン再現度、閲覧性

### フェーズ6: 主要インタラクション実装(JS)
- 目的: 画面内の基本操作を体験可能にする
- 実施内容:
  - タブ切替、フィルタ切替、進捗表示
  - 画面内の状態切替
- 成果物:
  - `webui/app.js`
- レビュー評価の観点:
  - 操作性、挙動の分かりやすさ

### フェーズ7: シナリオ検証・レビュー評価
- 目的: 想定ユースケースでの検証と改善点抽出
- 実施内容:
  - 検索・承認・配信・相談のシナリオ確認
  - レビュー評価(数値)の記録
- 成果物:
  - 課題一覧
  - レビュー評価表
- レビュー評価の観点:
  - シナリオ完遂性、改善点の妥当性

### フェーズ8: 改善反映・本番移行要件整理
- 目的: 本番実装に向けた要件ギャップを整理
- 実施内容:
  - 課題反映
  - API/データ連携前提の整理
- 成果物:
  - 改善版プロトタイプ
  - 本番移行要件リスト
- レビュー評価の観点:
  - 本番移行の準備度、課題解決度

## 2. 本番環境の詳細開発フェーズ (全10フェーズ)

### フェーズ1: 本番要件確定
- 目的: 本番に必要な機能・非機能を確定
- 実施内容:
  - 要件の優先度確定
  - 権限/監査/配信要件の詳細化
- 成果物:
  - 要件定義の確定版
- レビュー評価の観点:
  - 要件の明確性、抜け漏れ

### フェーズ2: アーキテクチャ設計確定
- 目的: 本番運用に耐える構成を確定
- 実施内容:
  - システム構成・運用方針の確定
  - 仮想環境を使わない前提での構成整理
- 成果物:
  - アーキテクチャ文書
- レビュー評価の観点:
  - 実運用の適合性、拡張性

### フェーズ3: データ設計確定
- 目的: データモデルと検索設計を確定
- 実施内容:
  - エンティティ/メタデータ定義
  - 検索インデックス設計
- 成果物:
  - データ設計文書
- レビュー評価の観点:
  - 検索性、拡張性

### フェーズ4: API設計確定
- 目的: UIと連携するAPI仕様を確定
- 実施内容:
  - エンドポイント設計
  - 権限チェックと監査項目の定義
- 成果物:
  - API仕様書
- レビュー評価の観点:
  - 一貫性、実装可能性

### フェーズ5: バックエンド基盤実装
- 目的: ナレッジ基盤とワークフローを実装
- 実施内容:
  - ナレッジ管理、承認、配信の実装
  - 監査ログの記録処理
- 成果物:
  - API実装一式
- レビュー評価の観点:
  - 機能充足、ログ整合

### フェーズ6: 検索・通知機能実装
- 目的: 検索精度と通知配信を担保
- 実施内容:
  - 検索インデックス構築
  - 通知配信処理の実装
- 成果物:
  - 検索/通知機能
- レビュー評価の観点:
  - 応答性、通知の確実性

### フェーズ7: WebUI統合実装
- 目的: API連携済みの画面を実装
- 実施内容:
  - UIのAPI連携
  - 権限制御に応じた表示
- 成果物:
  - 本番UI一式
- レビュー評価の観点:
  - 操作性、権限制御の正確性

### フェーズ8: セキュリティ/権限/監査強化
- 目的: 本番運用のセキュリティ基準を満たす
- 実施内容:
  - RBAC強化
  - 監査ログの保全方針適用
- 成果物:
  - セキュリティ確認結果
- レビュー評価の観点:
  - 監査/権限の妥当性

### フェーズ9: 品質保証・受入テスト
- 目的: 受入基準を満たすことを確認
- 実施内容:
  - 機能/非機能テスト
  - 受入テストの実施
- 成果物:
  - テスト結果報告
- レビュー評価の観点:
  - 不具合の収束度、基準達成度

### フェーズ10: 展開準備・運用引継ぎ
- 目的: 安定稼働のための引継ぎ完了
- 実施内容:
  - 展開計画/移行計画の確定
  - 運用手順/SLAの最終化
- 成果物:
  - 展開・運用ドキュメント
- レビュー評価の観点:
  - 運用準備の完了度
